# 项目协作说明

本项目是轻量级 Web 项目，后端接口使用 Node.js，不使用数据库。协作时优先保持实现简单、依赖少、改动范围小。

## 1. 项目定位

- 项目目标：从截图中提取字幕相关内容。
- 后端：Node.js API 服务，负责接口、文件接收、参数校验、业务编排和外部能力调用。
- 前端：负责页面交互、文件选择/上传、结果展示和必要的本地状态。
- 数据持久化：默认不接入数据库；如需保存结果，优先使用临时文件、导出文件或浏览器本地状态，并在需求中明确生命周期。

## 2. 推荐项目结构

实际目录可按现有代码调整，但职责应保持清晰。

```text
project/
  server/                  Node.js 接口服务
    src/
      app.js               应用入口或服务初始化
      routes/              API 路由
      services/            业务逻辑
      libs/                通用工具、外部服务封装
      middleware/          中间件
    uploads/               本地临时上传目录，禁止提交真实文件
    .env.example           环境变量模板
  frontend/                前端项目
    src/
      api/                 请求封装
      components/          通用组件
      views/               页面
      assets/              静态资源
  docs/                    项目说明、联调说明、重要变更记录
```

如果使用其他目录名，例如 `api/`、`backend/`、`client/`，继续沿用现有命名，不为了规范本身做无意义迁移。

## 3. 开发原则

- 优先遵循现有代码风格，不为小功能引入大型框架、ORM、队列或复杂抽象。
- 修改前先确认影响范围：接口、前端调用、文件处理、配置、日志和部署。
- 修复问题先定位原因，再做最小修正；不要用硬编码掩盖输入、路径、编码或异步时序问题。
- 同一需求涉及的接口、页面、配置和文档要同步处理。
- 不要提交真实上传文件、临时文件、构建产物、日志、密钥或本地缓存。

## 4. Node.js 接口约定

后端推荐保持简单分层：

```text
route/controller -> service -> libs
```

各层职责：

- `route/controller`：处理路由、参数读取、基础校验、响应封装。
- `service`：处理业务流程，例如图片解析、字幕提取、结果整理。
- `libs`：封装文件系统、OCR/模型调用、格式转换等通用能力。
- `middleware`：处理跨域、上传、错误处理、请求日志等横切逻辑。

接口统一返回稳定 JSON：

```json
{
  "code": 200,
  "message": "成功",
  "data": {}
}
```

接口规范：

- 统一 API 前缀，例如 `/api` 或 `/api/v1`，前端只维护一套 baseURL。
- 上传接口必须明确字段名、文件类型、大小限制和返回结构。
- 参数校验失败返回清晰错误，不把内部异常、完整路径、密钥或外部服务原始敏感信息暴露给前端。
- 耗时接口应有超时、失败提示和必要日志；避免请求无响应。
- 不使用数据库相关目录和概念，例如 `repository`、`dao`、`model`、`sql`、迁移脚本。

## 5. 无数据库约定

- 默认不新增数据库、ORM、迁移文件或持久化表结构。
- 不要为了保存少量状态引入数据库；优先考虑前端状态、临时文件或用户主动导出。
- 本地临时文件必须有清理策略，文件名使用安全字符，避免中文、空格和特殊符号导致路径或访问问题。
- 如未来确实需要持久化，先在需求或迭代记录中说明原因、数据范围、生命周期和替代方案，再引入存储方案。

## 6. 前端规范

- `src/api/` 统一封装请求，不在页面中散落大量 URL 和请求细节。
- 页面数据来自当前上传、接口响应、URL 参数或本地状态，不硬编码业务 ID。
- 上传、处理中、成功、失败、空结果等状态要有明确展示。
- 轮询或重试必须有停止条件，避免页面长期无意义请求。
- 前端展示字段应与后端返回结构对齐，不用假数据掩盖接口问题。
- 构建产物如 `dist/`、`.vite/`、`node_modules/` 不应提交。

## 7. 配置与密钥

- 真实配置放在 `.env` 或本地配置文件中，并加入 `.gitignore`。
- 仓库保留 `.env.example`，说明必要环境变量、端口和外部服务配置。
- API Key、访问密钥、模型密钥、云服务密钥不得提交。
- 本地、测试、生产环境使用不同配置，不把生产配置写入代码。
- 配置变更影响启动命令、代理、端口或外部服务时，同步更新文档。

## 8. 文件处理与外部服务

- 上传接口只负责接收、校验、临时保存和返回处理结果。
- 支持的图片格式、大小限制、临时目录和错误返回必须明确。
- 调用 OCR、模型或其他外部服务时要处理超时、失败、重试和错误摘要。
- 外部服务返回结构不稳定时应做容错解析，但不能吞掉关键错误。
- 临时文件处理完成后应尽量清理；需要保留时必须说明保留原因和位置。

## 9. 日志与错误处理

- 接口错误返回稳定结构，日志中保留必要上下文。
- 日志应能看出请求入口、处理步骤、成功或失败原因。
- 噪声日志应可通过配置关闭，关键错误不能被完全隐藏。
- 错误信息面向用户时保持简洁，面向日志时保留可定位问题的摘要。

## 10. 本地运行与端口

- 文档中明确后端启动命令和前端启动命令。
- 前端代理路径必须与后端 API 前缀一致。
- 如果本地端口被占用，先确认占用进程是否属于当前项目；属于当前项目时优先复用，不属于时再换端口或询问处理。
- 生产环境反向代理要保留 API 前缀，避免路径被错误截断。

示例：

```powershell
cd server
npm install
npm run dev

cd frontend
npm install
npm run dev
```

## 11. 验证要求

不要为了很小的改动频繁跑完整构建。优先选择和改动范围匹配的检查：

- Node.js 接口小改动：运行相关文件语法检查、相关接口冒烟测试或相关单元测试。
- 前端小改动：相关页面预览、类型检查或局部测试。
- 上传/文件处理改动：至少验证正常文件、非法格式、过大文件和外部服务失败路径。
- 公共能力或跨模块变更：再考虑完整测试或完整构建。

常见命令示例：

```powershell
npm run lint
npm run test
npm run build
```

## 12. 重要迭代记录

满足任一条件时，应在 `docs/iterations/` 新增或更新迭代记录：

- 改接口请求、响应字段、错误码或兼容行为。
- 改配置、权限、文件处理策略、上传限制或外部服务调用方式。
- 影响前端主要用户流程。
- 修复线上问题，且原因、影响或回滚方式值得追溯。
- 新增、删除或重构重要模块、provider 或 adaptor。

记录命名建议：

```text
docs/iterations/YYYY-MM-DD-short-topic.md
```

记录内容至少包含：

- 背景：为什么改。
- 变更内容：改了什么。
- 接口/兼容性影响：对调用方有什么影响。
- 配置/文件影响：是否需要新增配置或清理临时文件。
- 验证：做过哪些检查。
- 回滚方式：出问题时如何恢复。

## 13. Git 与交付

- 不要提交真实配置、本地缓存、构建产物、日志、临时上传文件。
- 不要回滚他人或用户已有改动，除非得到明确要求。
- 提交前检查改动范围，确保没有无关文件混入。
- 最终说明应包含：改了哪些文件、关键行为变化、做过哪些验证、哪些验证未执行及原因。
- 如果用户要求代码评审，优先说明风险、缺陷和缺失测试，再给变更摘要。
